CITADEL Design System Manual

A practical product manual for a sovereign city operating console: shared shell, differentiated module contexts, rendered component examples, source-backed color logic, and implementation-ready UI guidance.

ProductCITADEL - city governance, spatial intelligence, building control, construction, and utility operations.
ModulesCore, SPINE, BDCE, CMS, Citi Utility Manager.
Palette ruleModule colors identify context. Semantic colors identify state.
Token referenceImplementation tokens below

1. Brand Identity

CITADEL should feel like a sovereign city operating console: precise, evidence-led, modular, official, and modern. It should not feel like a generic SaaS dashboard or a decorative smart-city brochure.

Design thesisAuthority comes from structure, evidence, legibility, and restraint. Module color is used for orientation, not decoration.
CCITADELSovereign spatial systems

CITADEL is the platform identity. SYNTEGRA-SSS is the delivery consortium/company. The visual system must make CITADEL feel like the operating layer for land, buildings, infrastructure, services, and governance.

  • Evidence-led - every screen must explain source, confidence, version, owner, and next action.
  • Modular - users should know which engine they are using without losing the shared platform shell.
  • Official - permits, certificates, baselines, and receipts must look issued, auditable, and controlled.
  • Operational - CITADEL is for repeated daily work, not a landing-page mood board.
Do

Use restrained surfaces, clear labels, compact density, and module accents only where they orient the user to an active engine or selected evidence.

Do not

Use large decorative color washes, ambiguous dashboard cards, or AI-generated text that looks like an issued statutory decision.

4. Typography System

CITADEL uses compact, legible product typography. Display type should mark page identity; operational screens should prioritize labels, numbers, tables, and evidence.

Density ruleReserve large type for page identity. Workspaces need scan-friendly headings, compact rows, and tabular numerals.
Plus Jakarta Sans
Display and high-emphasis UI. Use for page titles, module names, and important numerical summaries.
Inter fallback
Body, navigation, form labels, table content, cards, and dense operational text.
JetBrains Mono
Tokens, codes, references, API-like IDs, parcel IDs, rule IDs, invoice IDs, and audit metadata.

5. Layout, Spacing, and Motion

The manual uses a practical documentation structure: sticky top header, persistent left navigation, and a fluid content column. Product screens should use the same calm density.

Interaction ruleMotion should clarify state changes only: hover, focus, tab switch, drawer, model selection, map layer selection, toast entry.
4px - icon gap, badge compression, tight table metadata.
8px - inline element gap, nav row icon spacing, chip padding.
12px - compact card padding, form input padding, row padding.
16px - standard panel padding and component preview padding.
24px - component inventory spacing and section group rhythm.
48px - section-to-section vertical rhythm in documentation pages.

Shape, Borders, Elevation, and State

--radius-xs 4px
Micro controls and dense tokens
Use for color-code badges, tiny swatches, table metadata, and small form adornments.
--radius-sm 6px
Controls and compact inputs
Use for buttons, text inputs, selected navigation rows, and small operational previews.
--radius-md 8px
Panels and component frames
Use for evidence cards, tables, map previews, workbench frames, and repeated item cards.
--radius-lg 12px
Only for brand marks and rare emphasis
Avoid large soft SaaS cards. Use this radius only where a distinctive object needs extra presence.
--line
Default border is neutral and one pixel
Use module color for selected context, focus, or evidence ownership, not for normal card borders.
--shadow-sm
Quiet lift for interactive surfaces
Use on selected rows, popovers, compact cards, and previews. Page sections should remain unframed.
--shadow-md
Reserved for workbench shells and overlays
Do not stack medium shadows inside cards. One elevation layer should explain hierarchy.
--focus-ring
Visible module-aware focus
Keyboard focus uses the active module ring. Error focus may add danger text, but the ring stays consistent.
20 / 36 / 40 / 44
Density ladder for controls
Badges are 20px, compact buttons 28px, standard buttons 36px, inputs 40px, touch targets at least 44px.

Motion Timing

Hover/focus
100-120ms ease for color, background, border, and shadow changes.
Drawer or inspector
180-220ms ease-out using opacity and transform, not layout-shifting width animations.
Map/model selection
120-160ms highlight transition. Respect reduced motion and keep evidence labels static.

Component API Specifications

Each component in the inventory has a defined API surface: tokens it consumes, states it supports, and behavior rules. These specs are the contract between design, frontend, and backend.

Button
Variants: primary, secondary, danger, official, compact, icon
Tokens: --btn-height (36px default, 32px compact), --btn-radius, --btn-font
States: enabled, hover, active, disabled, loading (with spinner)
Behavior: Primary uses --active-module or --citadel-core. Danger uses --semantic-danger. Official issuing actions use --citadel-core. Compact variant for table/inline actions. Never show two primary buttons in the same workspace region.
Backend: Each button maps to a single API endpoint. Disabled state implies POST /action is not available. Loading state prevents double submission.
Status Badge
Variants: success, warning, danger, attention, info, neutral
Tokens: --badge-height (20px), --badge-radius, --badge-font (10px, uppercase)
States: Always has a text label. Optional icon prefix. Never empty.
Behavior: Semantic state always wins over module identity. Never use module green for "passed" or module orange for "warning". Badges are smaller than buttons. Badges do not have hover or active states — they are not interactive.
Backend: Badge value maps to an enum field in the API response. The enum determines both the text and the color. No separate endpoint for badge rendering.
Form Field
Variants: text, search, select, date, file, textarea, radio, checkbox
Tokens: --input-height (40px), --input-radius, --input-border, --focus-ring
States: default, focused, filled, error, disabled, required
Behavior: Labels are always visible. Placeholder is never a substitute for a label. Error text appears below the field with --semantic-danger text color. Required marker uses --semantic-danger asterisk. Module-colored focus ring only for active context. Keyboard tab order follows visual order.
Backend: POST /submit receives { field: value }. Validation errors return { field: ["error message"] }. Backend never returns HTML error messages — always structured field-level errors.
Evidence Card
Variants: rule result, inspection finding, payment reconciliation, model object
Tokens: --card-radius, --card-shadow, --card-padding
States: default, expanded (row detail), selected (checkable card)
Behavior: Provenance is mandatory: source, owner, version, confidence, status, next action. Module accent identifies evidence owner; status badge identifies result. Evidence cards can stack vertically or appear in a 2-column grid. Never truncate the finding description.
Backend: Evidence endpoint returns { id, source, owner, version, confidence, status, next_action, body, attachments[] }. Attachments include download URL and thumbnail.
Data Table
Variants: standard, sortable, selectable, expandable, filtered
Tokens: --table-header-bg, --table-row-hover, --table-radius
States: default, hover, selected, expanded (row detail), empty, loading (skeleton)
Behavior: First column is always a persistent identifier. Status column uses semantic badges. Numeric values right-align. Row actions appear on hover. Extra columns collapse before horizontal overflow. Minimum 44px row height (36px for dense mode).
Backend: GET /resources?page=1&per_page=20&sort=updated_at:desc&filter[status]=issued. Response: { data: [], total: number, filtered_total: number, page: number, per_page: number, sort: string }. Never paginate without total and filtered_total.
Map Layer Row
Variants: parcel, proposal, conflict, utility corridor, inspection zone
Tokens: Module accent for ownership, semantic color for status, --layer-stroke, --layer-fill-opacity
States: visible, hidden, solo (isolated), loading (tile loading)
Behavior: Every layer needs color + stroke + pattern + legend label. Fill alone is never sufficient. Baseline parcels use solid stroke, proposals use dashed, conflicts use hatched pattern. Legend row always accompanies the layer list.
Backend: GET /map/layers returns { layers: [{ id, name, visible, stroke_style, fill_color, source, updated_at }] }. Tile endpoints follow XYZ schema. Layer toggles send PATCH /map/layers/{id} with visibility state.
AI Draft Block
Variants: rule interpretation, evidence summary, officer draft, compliance note
Tokens: --ai-surface, --ai-border, --ai-badge (attention semantic)
States: drafting (streaming), complete, edited, reviewed, approved
Behavior: AI output is always assistive. Must show citation, confidence, and audit status. Never visually match an issued decision. Actions are real buttons (Edit draft, Attach evidence). AI draft header says "Not issued decision".
Backend: POST /ai/draft with { context_id, prompt_type } returns { draft_id, body, citations[], confidence, status }. Draft save: PUT /ai/draft/{id}. Status changes: PATCH /ai/draft/{id}/status.
Payment Row
Variants: single charge, recurring, installment, refund
Tokens: Utility azure for ownership, semantic status for paid/pending/overdue/reconciling
States: paid, pending, reconciling, overdue, refunded, cancelled
Behavior: Amounts right-aligned with tabular figures. Receipt/source always visible. Reconciliation source (bank confirmation, manual entry) is never hidden. Payment actions (Pay now, Receipt, Dispute) appear on row hover.
Backend: GET /payments returns { payments: [{ id, account, charge, amount, status, receipt, source, due_date, paid_date }] }. Payment initiation: POST /payments/{id}/pay. Receipt download: GET /payments/{id}/receipt.
Callout / Alert
Variants: warning, danger, info, success
Tokens: Semantic token triple: --semantic-{type}, --semantic-{type}-bg, --semantic-{type}-border
States: visible, dismissible (with close action), persistent
Behavior: Inline notices appear inside the content context. Callouts use semantic triples with optional icon. The message must be explicit and actionable without color. Dismissible callouts save dismissal state per session. Never stack more than two callouts vertically.
Backend: Dismissible callouts send POST /user/dismissals with { callout_id }. Persistent callouts (system-level) are not dismissible and are controlled by GET /system/notices.

9. Accessibility Baseline

The stronger module palette was tested against white for body text. Accessibility also requires keyboard behavior, state text, reduced motion, and non-color affordances.

Contrast ruleAll module primaries pass WCAG AA body text contrast on white. Authority Gold Accent does not and is accent-only.
Color contrast
4.5:1 for body text
Core 15.82, SPINE 5.40, BDCE 7.93, CMS 5.18, Utility 4.87, Gold Text 5.93 on white.
Focus management
Visible focus on all controls
Use module focus rings for active workspaces; modals and drawers must trap focus and return it to the trigger.
Keyboard navigation
No mouse-only workflows
Tables, map layer lists, filters, tabs, and form actions must be reachable and operable by keyboard.
Color independence
Text, shape, and pattern required
Map layers and statuses must use labels, strokes, patterns, and badges. Color alone cannot carry legal meaning.
Reduced motion
Respect prefers-reduced-motion
Model and map animations must have static fallbacks. Avoid moving evidence labels during review.
Touch targets
Minimum 44px for controls
Module switchers, table actions, map layer toggles, and utility payment actions must keep reliable hit areas.

13. Navigation & Hierarchy

CITADEL has three navigation layers. Each answers a different user question. Mixing layers causes disorientation in high-pressure operational contexts.

Nav ruleGlobal actions never change module. Module actions never change work item. Work item actions never reorder the shell.

The Three Layers

1. Global shell navigationTenant switcher, module selector, global search, notifications, account menu. Live in the top bar. Always visible. Switching modules navigates to that module's default workspace.
2. Module navigationLeft sidebar containing the module's functional areas (e.g., SPINE: Layers, Baselines, Conflicts, Reports). Active section uses the module's --active-module accent. Persistent within a module session.
3. Work item navigationTabs, breadcrumbs, or step indicators within a single work item (permit, application, parcel, invoice). Work items nest inside module sections and close without affecting the module context.

Breadcrumb Pattern

Use breadcrumbs when the work item depth exceeds two levels (e.g., SPINE > Parcels > CC-A-104 > Conflicts). Breadcrumbs sit above the workspace heading and use --text-muted with --text-primary for the active leaf. Never use breadcrumbs as primary navigation — they are orientation only.

Navigation Anti-patterns

Do not

Put module-level navigation inside a page-level tab bar. Do not nest one module's sidebar inside another module's workspace. Do not use mega-menus, flyout navs, or hamburger menus for frequently accessed module functions.

Pre-Integration Checklist

  • Stage gate engine: each jurisdiction defines a stage template (e.g., foundation, structure, MEP, finishing, handover). Each stage has gate criteria (evidence requirements, inspection results, certifications) that must be satisfied before the stage can be certified.
  • Milestone tracking: milestones have planned date, actual date, status (pending, achieved, missed, re-baselined), evidence links, and owner. Delayed milestones trigger SLA notifications.
  • Inspection workflow: inspections are scheduled against a project or stage. Inspectors submit reports with structured findings, photos with geolocation, evidence documents, and an outcome (pass, conditional_pass, fail, re-inspection_required).
  • Certification chain: each stage certification records the certifying officer, their digital signature, evidence reviewed, conditions imposed, and the certification date. Certifications are durable and auditable.
  • Photo and evidence metadata: every site photo stores timestamp, GPS coordinates, device identifier, and photographer. Photos are linked to inspection findings and stage evidence requirements.
  • Handoff checklist: projects moving to operations must satisfy: all stages certified, outstanding corrections resolved, as-built documents submitted, utility accounts created, asset register updated, and warranty documents filed.
  • Contractor registry: contractors have registration number, specialisations, insurance expiry, prequalification status, portfolio links, and performance score. Only registered contractors can be assigned to projects.
  • Integration with BDCE: CMS reads approved permits from BDCE to create initial project structure. BDCE permit conditions flow into CMS stage gate requirements.
  • Integration with SPINE: CMS reads parcel and location context. Construction zone layers published to SPINE maps for situational awareness.
  • Stage timeline visualization: backend returns ordered stage list with status, dates, and evidence counts. The frontend renders this as a horizontal timeline with colour-coded stage states.

Pre-Integration Checklist

  • Payment gateway integration: at least one provider (Paystack, Flutterwave, Interswitch, or bank API) with card, bank transfer, and USSD payment methods. Gateway webhook endpoint at POST /api/utility/webhooks/payment for async payment confirmations.
  • Payment idempotency: every POST /api/utility/invoices/{id}/pay requires Idempotency-Key. Duplicate requests return the original payment response. Idempotency keys stored for 24 hours.
  • Receipt generation: receipts are generated as PDF with receipt number, account reference, payment amount, method, date, and reconciliation code. Receipts are stored in object store and served via GET /api/utility/payments/{id}/receipt.
  • Invoice lifecycle: draft → issued → due → overdue → paid → void. Invoices are generated on a schedule (monthly, quarterly, or on-demand depending on service type). Overdue invoices accrue late fees as defined in the jurisdiction settings.
  • Meter reading workflow: readings can be submitted by officers (on-site), by inspectors (during inspection), or via API integration (smart meters). Each reading stores value, timestamp, photo, GPS location, and reader identification.
  • Reconciliation engine: payments from the gateway are matched to open invoices. Manual payments (cash, cheque) require officer reconciliation with supporting evidence (bank statement reference, deposit slip). Unmatched payments are held in an suspense account.
  • Service request workflow: requests have type (new_connection, disconnection, fault_report, meter_reading, account_enquiry), priority (low, medium, high, critical), status (submitted, assigned, in_progress, resolved, closed), and SLA by priority level.
  • Integration with SPINE: utility accounts are linked to parcels via parcel_id. New property connections trigger SPINE layer updates. Utility corridor conflicts are flagged during BDCE permit review.
  • Integration with CMS: handoff from CMS creates the initial utility accounts for the property. Utility connection status is a handoff checklist item.
  • Financial reporting: daily settlement report, monthly revenue report, ageing analysis, and reconciliation report. Reports are generated as CSV and PDF, stored in object store, and accessible from the dashboard.

Integration Testing Requirements

  • Every module must pass integration tests against a real PostgreSQL instance (not mocked). Tests run in a transaction that is rolled back after each test.
  • Health endpoint test: verify GET /api/{module}/health returns 200 with all dependencies reported as healthy. Test with each dependency unavailable to verify correct degraded response.
  • Idempotency test: send the same POST request twice with the same Idempotency-Key. First returns 201, second returns 200 with Idempotent-Replayed: true header and identical body.
  • Authentication test: endpoints without valid JWT return 401. Endpoints with insufficient role return 403. Expired tokens return 401 with token_expired error code.
  • Pagination contract test: list endpoint without pagination params returns default page 1 with 20 results. total and filtered_total are always correct. per_page values above 100 are clamped to 100.
  • Error format test: every endpoint returns errors in the standard { error, detail, errors? } format. HTTP status codes match the error contract.
  • SSE event test: SSE stream connects with valid JWT. Events are received as JSON-encoded SSE messages. Reconnection after disconnect resumes from last event ID.
  • File upload test: upload a valid file returns { file_id, name, size, type, url }. Upload an invalid file type returns 422. Upload a file exceeding 50MB returns 413.
  • Cross-module integration test: create a parcel in SPINE → create a BDCE application referencing it → create a CMS project from the approved permit → verify the project reads correct parcel data from SPINE.
  • Performance test: list endpoint with 10,000 records responds in under 500ms (p95). Upload endpoint handles 10 concurrent 10MB uploads without timeout. Tile endpoint serves 100 req/s with < 200ms response time.
  • Rollback test: deploy a migration, run it, run the rollback, verify the schema returns to the pre-migration state with no data loss.
